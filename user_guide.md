# Architectural Q&A & User Manual

## Q1: What is the Agent Function-Call Training Data Generator, and what core problem does it solve?
### Answer:
The Agent Function-Call Training Data Generator is a pipeline designed to create high-quality, synthetic training datasets for fine-tuning Large Language Models (LLMs) on function-calling and tool-use tasks. In modern AI systems, agents are frequently required to interact with external APIs, databases, and local files by generating structured function calls (usually in JSON format) based on natural language instructions. However, generic datasets rarely cover the specific APIs, schema constraints, and complex multi-turn scenarios required for real-world enterprise deployments.

This project solves the data-scarcity problem by automatically generating large-scale, high-fidelity datasets containing natural user instructions, corresponding correct tool calls, and the full multi-turn conversations required to execute those calls. By combining two distinct LLMs (Groq for fast generation and Gemini for reasoning-heavy verification), the pipeline ensures the generated data is both syntactically correct and semantically aligned with user intent. It eliminates the need for manual dataset curation and allows developers to train small, specialized models (e.g., Llama-3-8B) to perform tool use with accuracy levels that rival or exceed frontier models.

## Q2: How does the dual-generator architecture (Groq and Gemini) work, and why was it chosen?
### Answer:
The project uses a dual-generator architecture where Groq (Generator A) and Gemini (Generator B) act as parallel data generation engines. Groq utilizes the ultra-fast LPU (Language Processing Unit) inference cloud to run open-source models like `llama-3.3-70b-versatile`, providing OpenAI-compatible function calling at high speeds. Gemini utilizes Google's Generative AI API to run models like `gemini-2.0-flash`, which features long context windows and JSON output modes.

This combination was chosen to maximize the diversity of the generated datasets and exploit the complementary strengths of each platform:
- **Groq (Llama 3.3)**: Generates highly idiomatic, fast function calls using standard system prompts and tool-use constraints. It provides raw speed and follows structured JSON schemas strictly.
- **Gemini (2.0 Flash)**: Offers advanced reasoning capabilities, serving as the primary validator and judge for subjective scoring (intent alignment and argument completeness).
By having both models generate solutions for the same instructions, we can create Direct Preference Optimization (DPO) pairs, comparing Groq's output directly with Gemini's output to find the optimal formatting and parameter choices.

## Q3: What is the 4-tier taxonomy of tool-use examples generated by this pipeline?
### Answer:
The pipeline classifies all training examples into four distinct structural categories, or "Types," to train models on different aspects of tool use:
1. **Type 1 (SINGLE)**: A straightforward, single-turn instruction that maps directly to a single tool call. For example, "What's the weather in Paris?" maps to `get_current_weather(location="Paris")`. This teaches the model basic extraction and schema mapping.
2. **Type 2 (CHAIN)**: A multi-turn instruction where the agent must call a sequence of tools sequentially, feeding the output of one tool as the input to the next. For example, "Find a free slot for tomorrow, then schedule a meeting with Sarah." This teaches the model sequential reasoning and context tracking.
3. **Type 3 (AMBIGUOUS)**: An instruction that is missing critical parameters or is otherwise unclear. Instead of guessing values, the model must respond by asking a clarifying question. This teaches the model boundary detection and conversational safety.
4. **Type 4 (PARALLEL)**: An instruction requiring multiple independent tool calls simultaneously. For example, "Check the stock prices for AAPL and MSFT." This teaches the model to generate multiple tool calls in a single turn.

## Q5: How is a tool schema defined in this project, and what is its role in the pipeline?
### Answer:
Every tool is defined as a standalone JSON Schema file located in the `schemas/<domain>/` directory. The schema follows the Draft-07 JSON Schema specification. It describes the tool's name, domain, description, parameter names, types, constraints (e.g., minimums, maximums, enums), and required fields.

The schema plays several vital roles:
- **API Definition**: Loaded into the `SchemaRegistry` and converted into OpenAI function tool definitions or Gemini tool declarations to instruct the models on what APIs are available during generation.
- **Verification Hook**: Used by the `Validator` to programmatically parse and check generated tool calls, flagging any incorrect types, missing required arguments, or out-of-bounds parameter values.
- **Scoring Reference**: Used by the `Scorer` to evaluate parameter correctness and identify hallucinated arguments.

## Q6: How does the seed instruction system work, and how does the paraphraser expand it?
### Answer:
The pipeline begins with 600 human-authored "seed instructions" (50 per domain × 12 domains) saved in `instructions/*.json`. Each seed specifies the raw instruction text, the example type, the difficulty, and the expected tool calls.

To generate a large-scale training set, the `InstructionParaphraser` uses the Gemini API to expand these seeds. It reads each seed instruction and generates 10 unique, natural variations (paraphrases) while preserving the core intent, example type, and expected tool calls. The paraphrased variants vary in tone (casual, professional, terse), phrasing, and complexity. This transforms the 600 seeds into a diverse pool of 6,000 unique instructions, preventing model overfitting and exposing the agent to varied natural language structures.

## Q7: How does the embedding-based deduplication system prevent duplicate instructions?
### Answer:
To ensure dataset diversity and quality, the pipeline implements an embedding-based deduplication stage. When new instructions are paraphrased, they are passed to Google's `text-embedding-004` model to compute a dense vector representation.

These vectors are stored in the database. When a new instruction is processed, its cosine similarity is calculated against all existing instructions in the same domain. If the cosine similarity exceeds the `DEDUP_SIMILARITY_THRESHOLD` (default: 0.95), the instruction is rejected as a duplicate and discarded. This ensures that paraphrased variants are semantically unique, preventing the model from memorizing redundant instructions.

## Q8: What tables make up the PostgreSQL database schema, and what is stored in each?
### Answer:
The PostgreSQL database consists of 5 main tables:
1. `instructions`: Stores the paraphrased instruction pool (ID, text, domain, example type, difficulty, seed ID, and embedding vector).
2. `examples`: Stores the generated examples (ID, instruction ID, instruction text, domain, type, difficulty, generator, tool calls, conversation JSON, and validation status).
3. `scores`: Stores the detailed quality score for each example (composite score, quality tier, and individual axis scores for schema, completeness, alignment, hallucination, and coherence).
4. `preference_pairs`: Stores DPO preference pairs (ID, instruction ID, chosen example ID, rejected example ID, score delta, and chosen/rejected conversation histories).
5. `failures`: Logs all generation and validation failures for debugging and negative-sample training.

## Q9: Why does this project use SQLAlchemy Core instead of SQLAlchemy ORM?
### Answer:
SQLAlchemy Core was chosen over the traditional ORM (Object-Relational Mapper) layer for three primary reasons:
- **Performance**: Core operates closer to the database driver (psycopg2), avoiding the overhead of session management, object state tracking, and unit-of-work patterns. This is crucial when concurrently inserting thousands of complex records.
- **Pydantic Alignment**: The project already uses Pydantic v2 as its primary data modeling and validation layer. Using SQLAlchemy Core tables allows us to map raw database rows directly into Pydantic models (using `model_validate`) without maintaining a duplicate set of ORM classes.
- **Complex Query Control**: Operations like vector similarity queries (for deduplication) and custom joins (for preference pairs) are cleaner to write in SQLAlchemy Core's explicit SQL syntax.

## Q10: How does the 3-stage validation engine process generated tool calls?
### Answer:
The `Validator` class orchestrates three sequential validation checks:
1. **Schema Validation**: Checks if the generated tool name exists in the registry and validates the arguments using `jsonschema`. It catches syntax errors, missing fields, type mismatches, and out-of-bounds values.
2. **Semantic Validation**: Compares the user instruction with the description of the selected tools using text embeddings. If the similarity is below the threshold, it flags a semantic misalignment.
3. **Chain Validation (Type 2)**: Checks if parameter values in multi-turn examples are passed correctly across turns (e.g., if a tool output from turn 1 is referenced correctly in turn 2).

## Q11: How does the 5-Axis Scoring Rubric evaluate generated training data?
### Answer:
The Scorer evaluates examples across five distinct criteria:
1. **Schema Correctness (Weight: 25%)**: Evaluates syntax correctness. Computed programmatically by the validator.
2. **Argument Completeness (Weight: 20%)**: Evaluates if the agent extracted all available parameters from the instruction. Judged by Gemini.
3. **Intent Alignment (Weight: 25%)**: Evaluates if the generated calls satisfy the user's request. Judged by Gemini.
4. **Hallucination Score (Weight: 20%)**: Programmatic check. Scores 1.0 if all tool names and parameters exist in the registry, 0.0 otherwise.
5. **Chain Coherence (Weight: 10%)**: Programmatic check for Type 2 examples. Evaluates if multi-turn variables are threaded correctly.

---

## Q12: How is the composite score computed, and how are quality tiers assigned?
### Answer:
The composite score is calculated as a weighted sum of the five axis scores:
$$\text{Composite} = 0.25 \times \text{Schema} + 0.20 \times \text{Completeness} + 0.25 \times \text{Alignment} + 0.20 \times \text{Hallucination} + 0.10 \times \text{Coherence}$$

Based on the composite score, the example is assigned to a quality tier:
- **High Quality (Tier: High)**: Composite score $\ge 0.85$. These are the primary positive examples used for fine-tuning.
- **Medium Quality (Tier: Medium)**: Composite score between $0.65$ and $0.84$. These are valid but sub-optimal calls.
- **Low Quality (Tier: Low)**: Composite score $< 0.65$. These are rejected and written to the failures log.

## Q13: What is the LLM-as-a-Judge pattern, and where is it applied in the scorer?
### Answer:
The "LLM-as-a-Judge" pattern refers to using a capable language model (Gemini 2.0 Flash) to evaluate subjective aspects of another model's output against a scoring rubric.

In `src/scorer.py`, this pattern is applied to evaluate **Argument Completeness** and **Intent Alignment**. Programmatic checks can verify if a JSON payload is valid, but they cannot assess if the model made a logical guess or missed a subtle parameter from a natural language request. The scorer sends the user's instruction, the JSON schemas of all tools in the domain, and the generated tool call to Gemini. Gemini returns a structured JSON response containing scores between 0.0 and 1.0 and detailed reasoning for each axis.

## Q14: How are DPO (Direct Preference Optimization) preference pairs constructed?
### Answer:
DPO requires pairs of chosen (preferred) and rejected (disliked) model responses for the same prompt. The `PreferencePairBuilder` constructs these pairs by:
- Identifying instructions for which both Groq and Gemini generated valid examples.
- Retrieving the composite scores for both examples.
- Confirming that the difference between the scores (score delta) is at least `0.05` to ensure a clear preference signal.
- Formatting the chosen and rejected outputs into the standard conversation structures (system prompt, user query, tool call assistant turn) and saving them in the `preference_pairs` table.

---

## Q15: How are Failure Modes categorized, and why are they stored in the database?
### Answer:
When an example fails validation or receives a low score, it is logged in the `failures` table along with a specific `FailureMode` enum value. These modes include:
- `HALLUCINATED_TOOL`: The model called a tool name that does not exist in the schema registry.
- `HALLUCINATED_PARAM`: The model generated a parameter key not defined in the tool's schema.
- `TYPE_MISMATCH`: The parameter value has the wrong data type.
- `MISSING_REQUIRED`: The model omitted a parameter marked as required by the schema.
- `ENUM_VIOLATION`: The model passed a value not present in the parameter's string enum constraints.
- `LOW_SEMANTIC_SIMILARITY`: The selected tools do not align with the instruction.
- `BROKEN_CHAIN`: The multi-turn variables were not passed correctly.

These failures are stored to provide developers with detailed metrics on where models fail most frequently, allowing them to adjust system prompts, schemas, or models.

## Q18: Why did we remap the host database port to `54321`?
### Answer:
By default, PostgreSQL databases listen on port `5432`. On Windows environments, it is common to have a native PostgreSQL instance running as a system service on port `5432` (and port `5433` if multiple versions are installed).

If our Docker Compose database container attempted to bind to port `5432` or `5433`, it would conflict with these host services. By mapping the Docker container's internal port `5432` to host port `54321` (i.e. `"54321:5432"`), we ensure the dockerized database can run alongside any native PostgreSQL instances without port conflicts or authentication errors.

## Q23: How does the pipeline handle rate limits and API connection drops?
### Answer:
The pipeline uses the `tenacity` library to handle rate limits (HTTP 429) and network drops. All raw API call methods (in `generator.py` and `scorer.py`) are decorated with `@retry`.

If an API call fails due to a rate limit or transient network error, `tenacity` waits using an exponential backoff strategy (e.g., waiting 2 seconds, then 4 seconds, then 8 seconds) before retrying the call. The pipeline allows up to 3 retries per sample before logging an `API_ERROR` in the database and moving to the next instruction.

## Q26: What is a sequential chain tool use case (Type 2), and how does the validator check it?
### Answer:
A sequential chain tool use case (Type 2) requires the model to execute a sequence of dependent steps. For example:
1. `find_free_slot(attendees=["Bob"])` $\rightarrow$ returns `"2026-06-14T15:00:00Z"`.
2. `create_event(title="Meeting", start_time="2026-06-14T15:00:00Z", ...)` $\rightarrow$ uses the output from step 1.

The `ChainValidator` validates these examples by parsing the conversation history. It checks if the arguments of subsequent tool calls contain values returned by prior tool executions. If the model fails to reference the outputs of previous turns, the validator flags the example with a `BROKEN_CHAIN` failure mode.

---

## Q27: What is an ambiguous tool use case (Type 3), and how should a model handle it?
### Answer:
An ambiguous tool use case (Type 3) occurs when a user request is missing parameters required to execute a tool call. For example, "Cancel my meeting" is ambiguous because the agent needs to know which meeting to cancel.

A model should handle this by:
- Identifying the missing required parameter (`event_id` in this case).
- Generating a natural language question asking the user for the missing details.
- Avoiding making tool calls with placeholder or guessed values.
The validator ensures these examples do not contain tool calls and that the assistant's turn is a clarifying question.

## Q28: How does the SchemaRegistry load schemas in under 1 millisecond?
### Answer:
The `SchemaRegistry` loads all schemas at startup by traversing the `schemas/` directory using Python's `pathlib.Path.rglob`. It reads each JSON file, parses it into a `ToolSchema` Pydantic model, and indexes it in two in-memory dictionaries:
- `_by_name`: A map of `tool_name` $\rightarrow$ `ToolSchema` for $O(1)$ lookups.
- `_by_domain`: A map of `domain` $\rightarrow$ list of `ToolSchema` to retrieve all tools in a domain.

Because JSON files are small, reading all 46 files takes less than 1 millisecond. This in-memory design avoids database overhead during validation.

## Q31: How does the generator prevent prompt injection and handle malicious inputs?
### Answer:
The pipeline mitigates prompt injection risks by separating user input from system instructions:
- **System Isolation**: System prompts (in `generator.py`) are passed as distinct developer/system messages in the API request payload, separated from the user instruction.
- **Strict Parameter Constraints**: The validator checks all generated parameters against JSON Schema constraints. If a model generates SQL injection strings or shell commands, they will fail schema type validations or be caught by database constraints.
- **Environment Isolation**: Database executions are run in isolated container environments, protecting the host system from malicious output execution.

## Q33: How does the multi-threaded worker pipeline parallelize API calls?
### Answer:

Since generation involves sending requests to external APIs (Groq and Gemini) and waiting for responses, execution is bound by network latency rather than CPU usage. Running multiple worker threads allows the pipeline to execute multiple API calls concurrently. While one thread waits for a response from Groq, other threads handle validation, scoring, or database writes, improving generation throughput.

## Q35: How does the validator handle type coercion (e.g., strings to integers)?
### Answer:
The validator uses the `jsonschema` library, which enforces strict type checking by default. If a schema specifies a parameter as an integer (e.g., `3`), but the model returns it as a string (e.g., `"3"`), the validation check will fail with a `TYPE_MISMATCH` error.

This strict type checking is intentional. During fine-tuning, we want the model to learn to output values in the correct schema types directly, rather than relying on post-processing code to coerce and fix parameter types.

## Q39: What is semantic similarity check (Stage 2) in the validator, and how is it calculated?
### Answer:
The semantic similarity check verifies if the selected tool matches the user's intent. Programmatic schema validation only checks if a JSON payload is syntactically valid; it cannot tell if the model chose the wrong tool (e.g., using `delete_event` instead of `create_event`).

To check this, the validator:
- Retrieves the description of the called tool.
- Generates text embeddings for both the user instruction and the tool description using Google's `text-embedding-004` model.
- Calculates the cosine similarity between the two vectors.
If the similarity falls below the `EMBEDDING_SIMILARITY_THRESHOLD` (default: 0.75), the validator flags the call as semantically misaligned.

## Q40: What happens when the generator encounters a completely malformed JSON response?
### Answer:
If a generator model returns a response that cannot be parsed as valid JSON:
- The parser catches the `JSONDecodeError`.
- It registers the failure with the `FailureMode.MALFORMED_JSON` mode.
- The failure details, along with the raw model response, are logged in the `failures` table.
- The pipeline then moves to the next instruction in the queue.
This error handling prevents malformed API responses from stopping the pipeline execution.

## Q42: How does the pipeline handle multi-turn conversations for agent training?
### Answer:
For Type 2 (Chain) multi-turn examples, the pipeline builds conversation histories using the `ConversationTurn` Pydantic model.

Each step in the execution chain is represented as a turn:
- **User Turn**: Contains the initial instruction.
- **Assistant Turn**: Contains the model's generated tool call.
- **Tool Turn**: Simulates the tool output (e.g., database records or calendar slots).
- **Final Assistant Turn**: Synthesizes the final response for the user.
The resulting conversation history teaches the model to follow multi-turn flows during training.

## Q43: How does the scoring judge handle ambiguous cases (Type 3) where no tool calls are expected?
### Answer:
For Type 3 (Ambiguous) instructions, the model is expected to ask a clarifying question instead of making a tool call.

The scorer handles this case by:
- Checking if the instruction type is `Type 3` and no tool calls were generated.
- Directing the Gemini judge to evaluate if the model's response is a helpful clarifying question.
If the judge determines the question is specific and correctly identifies the missing information, it assigns a score of `1.0` for Intent Alignment and Argument Completeness.

## Q44: What are the security benefits of running the PostgreSQL database in Docker?
### Answer:
Running the database in a Docker container provides several security and configuration benefits:
- **Isolation**: The database process and storage are isolated from the host operating system, preventing accidental data loss or security leaks.
- **Consistency**: Docker ensures the database runs in the same environment (Alpine Linux with PostgreSQL 16) regardless of the host OS (Windows, macOS, or Linux).
- **Clean Resets**: Developers can delete and recreate the database volume with a single command (`docker compose down -v`), ensuring a clean state for testing.

## Q47: Why does the project use the Pydantic v2 `model_config` instead of the legacy Pydantic v1 `Config` class?
### Answer:
The project uses Pydantic v2 configurations (`model_config = ConfigDict(...)`) to leverage the performance improvements in Pydantic v2.

Pydantic v2's core validation engine is written in Rust, making it significantly faster than v1. This performance difference is valuable when validating thousands of nested tool schemas and conversation histories during large pipeline runs.

## Q49: What is the purpose of the `FailureRecord` model, and where are failures logged?
### Answer:
The `FailureRecord` model is a Pydantic class that logs validation and generation failures:
- **Properties**: Captures the instruction ID, the target generator, the specific `FailureMode`, error details, and the raw API response.
- **Storage**: Written to the `failures` table in the PostgreSQL database.
Logging these failures helps developers identify common model errors, such as hallucinated parameters or incorrect types, to refine schemas and prompts.
